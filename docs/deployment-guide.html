Prerequisites

Before deploying gif-captcha in production, ensure you have:

  • Node.js 18+ (LTS recommended; 20.x or 22.x preferred)
  • npm 9+ or yarn 1.22+
  • At least 256 MB RAM per instance (512 MB recommended for high traffic)
  • Disk space for session data, audit logs, and GIF challenge assets
  • A reverse proxy (nginx, Caddy, or cloud load balancer) for TLS termination

Note: gif-captcha is a library — it runs embedded in your application server. These instructions cover deploying an app that uses gif-captcha, not a standalone service.

Environment Configuration

gif-captcha reads configuration through its ConfigValidator. For production, set these via environment variables or a config file:

VariableDescriptionDefaultProduction Recommendation
GIF_CAPTCHA_SECRET HMAC secret for session tokens none (required) 32+ byte random string. Rotate quarterly.
GIF_CAPTCHA_TTL Challenge expiry in seconds 300 120–300s depending on UX needs
GIF_CAPTCHA_MAX_ATTEMPTS Max solve attempts per session 5 3–5; lower for high-security flows
GIF_CAPTCHA_DIFFICULTY Base difficulty level (1–10) 5 5–7; adaptive-difficulty-tuner adjusts per-session
GIF_CAPTCHA_RATE_LIMIT Requests per minute per IP 30 10–30 depending on traffic pattern
GIF_CAPTCHA_LOG_LEVEL Logging verbosity info warn in prod; debug only for incident triage
GIF_CAPTCHA_WEBHOOK_URL Webhook endpoint for events none Set to receive solve/fail/anomaly events
NODE_ENV Node.js environment development Always set to production 
# Example .env for production
NODE_ENV=production
GIF_CAPTCHA_SECRET=your-32-byte-random-secret-here
GIF_CAPTCHA_TTL=180
GIF_CAPTCHA_MAX_ATTEMPTS=3
GIF_CAPTCHA_DIFFICULTY=6
GIF_CAPTCHA_RATE_LIMIT=20
GIF_CAPTCHA_LOG_LEVEL=warn

Security: Never commit GIF_CAPTCHA_SECRET to version control. Use your platform's secrets manager (AWS Secrets Manager, Vault, Kubernetes Secrets, etc.).

Docker Deployment

The repository includes a multi-stage Dockerfile. Build and run:

# Build the image
docker build -t gif-captcha:latest .

# Run with environment variables
docker run -d \
  --name gif-captcha \
  -p 3000:3000 \
  -e NODE_ENV=production \
  -e GIF_CAPTCHA_SECRET="$(openssl rand -hex 32)" \
  -e GIF_CAPTCHA_LOG_LEVEL=warn \
  --memory=512m \
  --cpus=1 \
  --restart=unless-stopped \
  gif-captcha:latest

Docker Compose

For multi-service deployments with a reverse proxy:

      interval: 30s

      retries: 3

    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
      - /etc/letsencrypt:/etc/letsencrypt:ro
    ports:
      - "443:443"
      - "80:80"
    depends_on:
      captcha:
        condition: service_healthy

Reverse Proxy Setup

Always run gif-captcha behind a reverse proxy that handles TLS, compression, and rate limiting at the edge.

nginx

upstream captcha_backend {
    server 127.0.0.1:3000;
    keepalive 32;
}

server {
    listen 443 ssl http2;
    server_name captcha.example.com;

    ssl_certificate     /etc/letsencrypt/live/captcha.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/captcha.example.com/privkey.pem;

    # Security headers
    add_header X-Content-Type-Options nosniff always;
    add_header X-Frame-Options DENY always;
    add_header Referrer-Policy strict-origin-when-cross-origin always;
    add_header Content-Security-Policy "default-src 'self'; img-src 'self' data:; style-src 'self' 'unsafe-inline'" always;

    # Rate limiting at edge
    limit_req_zone $binary_remote_addr zone=captcha:10m rate=10r/s;

    location / {
        limit_req zone=captcha burst=20 nodelay;
        proxy_pass http://captcha_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Timeouts
        proxy_connect_timeout 5s;
        proxy_send_timeout 10s;
        proxy_read_timeout 30s;
    }

    # Cache static challenge assets
    location /assets/ {
        proxy_pass http://captcha_backend;
        proxy_cache_valid 200 1h;
        add_header Cache-Control "public, max-age=3600";
    }
}

Caddy (simpler alternative)

captcha.example.com {
    reverse_proxy localhost:3000 {
        header_up X-Real-IP {remote_host}
    }
    header X-Content-Type-Options nosniff
    header X-Frame-Options DENY
    rate_limit {remote.ip} 10r/s
}

Scaling Strategies

Horizontal Scaling

gif-captcha is stateless by default — session data is stored in-memory per instance. For multi-instance deployments:

  • Sticky sessions: Route the same client IP to the same instance (simplest). Configure at the load balancer level.
  • Shared session store: Plug a Redis or Memcached backend into the session layer so any instance can validate any token.
  • Stateless tokens: Use the HMAC-based token mode where the challenge answer is encrypted in the token itself — no server-side session needed.

Capacity Planning

Use the built-in captcha-capacity-planner module to estimate resource needs:

const { CapacityPlanner } = require('gif-captcha/src/captcha-capacity-planner');

const planner = new CapacityPlanner();
const estimate = planner.estimate({
    peakRequestsPerSecond: 500,
    avgChallengeSizeKB: 150,
    sessionTTLSeconds: 180,
    targetLatencyP99Ms: 200
});

console.log(estimate);
// { instances: 3, memoryPerInstanceMB: 384, bandwidthMbps: 60, ... }

Monitoring & Alerting

Key Metrics to Track

Use captcha-stats-collector and captcha-health-monitor to emit these metrics:

  • Solve rate: Percentage of challenges solved successfully. Normal: 85–95%. Below 70% may indicate difficulty is too high or a UX problem.
  • Bot detection rate: Challenges flagged by behavioral biometrics. Spikes indicate an attack wave.
  • Latency (p50/p95/p99): Challenge generation and verification response times.
  • Anomaly score: From captcha-anomaly-detector. Sustained scores > 0.7 warrant investigation.
  • Session volume: Active sessions. Used for capacity planning.
  • Rate limit triggers: How often the rate limiter fires. High counts = possible DDoS or scraping.

Webhook Integration

Configure webhook-dispatcher to push events to your monitoring stack:

const { WebhookDispatcher } = require('gif-captcha/src/webhook-dispatcher');

const dispatcher = new WebhookDispatcher({
    url: process.env.GIF_CAPTCHA_WEBHOOK_URL,
    events: ['solve.success', 'solve.failure', 'anomaly.detected', 'rate.limited'],
    batchSize: 10,
    flushIntervalMs: 5000
});

Health Checks

Expose a /health endpoint for load balancers and orchestrators. The captcha-health-monitor module provides a comprehensive check:

const { HealthMonitor } = require('gif-captcha/src/captcha-health-monitor');

const monitor = new HealthMonitor();

app.get('/health', (req, res) => {
    const status = monitor.check();
    res.status(status.healthy ? 200 : 503).json(status);
});

// Response format:
// {
//   "healthy": true,
//   "uptime": 86400,
//   "checks": {
//     "challengePool": { "status": "ok", "available": 150 },
//     "sessionStore": { "status": "ok", "activeSessions": 42 },
//     "memory": { "status": "ok", "usedMB": 128, "limitMB": 512 }
//   }
// }

Kubernetes: Use /health for both liveness and readiness probes. Set initialDelaySeconds: 5 and periodSeconds: 15.

Backup & Recovery

What to Back Up

  • Challenge assets (GIFs): Store in a versioned S3 bucket or equivalent. These are your core content.
  • Configuration: Keep in version control or secrets manager. Never in the container image.
  • Audit logs: From captcha-audit-log. Ship to a log aggregator (ELK, Loki, CloudWatch).
  • Trust score data: From trust-score-engine. Export periodically if you need historical trust analysis.

Recovery Procedure

  1. Deploy a fresh instance with the same environment configuration.
  2. Restore challenge assets from backup to the assets directory.
  3. The challenge pool, session data, and trust scores will rebuild automatically from live traffic.
  4. Verify with /health endpoint and check solve rates return to normal within 5–10 minutes.

Operational Runbooks

🚨 Bot Attack Wave

  1. Check captcha-anomaly-detector dashboard for attack pattern.
  2. Increase base difficulty: GIF_CAPTCHA_DIFFICULTY=8
  3. Lower rate limit: GIF_CAPTCHA_RATE_LIMIT=5
  4. Enable enhanced biometrics in behavioral-biometrics config.
  5. Review fraud-ring-detector output — block identified IP ranges at the edge (nginx/WAF).
  6. Monitor solve rate; relax settings once attack subsides.

📉 Solve Rate Drop

  1. Check if a recent deployment changed difficulty or challenge content.
  2. Review adaptive-difficulty-tuner logs — it may have over-corrected.
  3. Test challenges manually in a browser.
  4. Check challenge-diversity-analyzer — low diversity can cause repetitive hard challenges.
  5. Temporarily lower GIF_CAPTCHA_DIFFICULTY and redeploy.

💾 High Memory Usage

  1. Check active session count via /health.
  2. Reduce GIF_CAPTCHA_TTL to expire sessions faster.
  3. Check for session leaks — captcha-session-replay keeps session data; disable replay in production if not needed.
  4. Scale horizontally to distribute load.
  5. Restart instances if memory doesn't recover (Node.js GC may need a nudge under extreme fragmentation).
