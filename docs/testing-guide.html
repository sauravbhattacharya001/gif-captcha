Testing Guide

Running, writing, and maintaining the gif-captcha test suite

Overview

gif-captcha uses Node.js's built-in node:test runner — no external test frameworks required. The test suite comprises 103 test files covering all 49 source modules plus integration, edge-case, and legacy compatibility scenarios.

Running Tests

Full Suite

npm test
# Equivalent to: node --test tests/*.test.js

With Coverage

# Text + LCOV reports
npm run test:coverage

# Check coverage thresholds pass
npm run coverage:check

Single File

# Run one test file
node --test tests/trust-score-engine.test.js

# Run with verbose output
node --test --test-reporter=spec tests/fraud-ring-detector.test.js

Pattern Matching

# Run all captcha-* tests
node --test tests/captcha-*.test.js

# Run all bot-related tests
node --test tests/bot-*.test.js
Use --test-only to run only tests marked with { only: true } during development.

Writing Tests

Test File Structure

const { describe, it, beforeEach } = require('node:test');
const assert = require('node:assert/strict');
const { MyModule } = require('../src/my-module.js');

describe('MyModule', () => {
    let instance;

    beforeEach(() => {
        instance = new MyModule({ /* config */ });
    });

    describe('constructor', () => {
        it('applies default configuration', () => {
            assert.strictEqual(instance.enabled, true);
        });

        it('validates required options', () => {
            assert.throws(
                () => new MyModule({}),
                { message: /required/ }
            );
        });
    });

    describe('process()', () => {
        it('handles normal input', () => {
            const result = instance.process({ data: 'test' });
            assert.ok(result.score >= 0 && result.score <= 1);
        });

        it('handles edge cases gracefully', () => {
            assert.doesNotThrow(() => instance.process(null));
            assert.doesNotThrow(() => instance.process(undefined));
        });
    });
});

Conventions

  • File naming: tests/{module-name}.test.js mirrors src/{module-name}.js
  • Assertions: Use node:assert/strict — strict equality by default
  • No mocks framework: Use manual stubs/fakes where needed (keep tests dependency-free)
  • Self-contained: Each test file must run independently — no shared state between files
  • Descriptive names: it('returns empty array when no sessions match criteria')
  • Edge cases first: Test null/undefined/empty inputs before happy paths

What to Test

CategoryWhat to cover
ConstructionDefault config, custom config, invalid config (throws)
Core logicHappy path, boundary values, edge cases
ValidationType coercion, clamping, null handling
MemoryLRU eviction, TTL expiry, max-size bounds
ScoringScore ranges [0,1], decay over time, weight normalization
Export/ImportRound-trip serialization, corrupt data handling
ConcurrencySession isolation, no cross-contamination
Never use setTimeout or real timers in tests. If testing time-based features, inject a clock function or manipulate timestamps directly.

Coverage

Coverage is tracked via c8 using V8's native instrumentation — no transpilation or Istanbul needed.

Viewing Reports

# Terminal summary
npm run test:coverage

# HTML report (opens in browser)
npx c8 --reporter=html node --test tests/*.test.js
open coverage/index.html

Thresholds

CI enforces minimum coverage via .c8rc.json:

  • Lines: 75%
  • Functions: 85%
  • Branches: 85%

If your changes drop coverage below thresholds, add tests for the new code paths. The coverage:check script will fail in CI if thresholds aren't met.

CI Integration

Tests run automatically on every push and PR via GitHub Actions (.github/workflows/ci.yml):

  • Matrix: Node.js 18, 20, 22
  • Steps: install → syntax-check → test → coverage
  • Coverage reports uploaded to Codecov
  • Branch protection requires CI pass before merge

Debugging Failing Tests

Verbose Output

# Show all test names (passing and failing)
node --test --test-reporter=spec tests/my-module.test.js

Isolate a Single Test

// Temporarily mark one test as `only`
it('the failing test', { only: true }, () => {
    // ...
});

// Run with --test-only flag
node --test --test-only tests/my-module.test.js

Common Failures

SymptomLikely causeFix
Tests pass alone, fail together Shared module-level state Reset in beforeEach or create fresh instances
Flaky timing assertions Using Date.now() directly Inject a clock/timestamp parameter
ERR_MODULE_NOT_FOUND Wrong relative path in require() Check ../src/ prefix
Score out of expected range Missing clamp after arithmetic Use Math.max(0, Math.min(1, score))

Performance Testing

The perf-caching.test.js and benchmark.test.js files cover performance-sensitive paths. When optimizing hot paths:

  • Measure before and after with performance.now()
  • Test with realistic data sizes (1000+ sessions, 10000+ events)
  • Verify O(n) vs O(n²) behavior doesn't regress
  • Check memory bounds — LRU caches should evict properly

Adding Tests for New Modules

  1. Create tests/{module-name}.test.js
  2. Follow the structure pattern above
  3. Cover: construction, core methods, edge cases, export/import if applicable
  4. Aim for 85%+ coverage on your new module
  5. Run npm run coverage:check to verify thresholds
  6. Test passes in isolation: node --test tests/{module-name}.test.js
Look at trust-score-engine.test.js or session-risk-aggregator.test.js as reference examples — they demonstrate thorough coverage of construction, scoring, decay, bounds, export/import, and session management patterns.
